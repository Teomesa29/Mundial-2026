# Justificación de Decisiones de Diseño

- **Uso de JSONB en `correct_answer` y Audit Log**: JSONB otorga una enorme flexibilidad. En las apuestas especiales, la respuesta correcta puede ser un ID de un equipo, un ID de jugador, un número entero (ej. goles totales) o un booleano (ej. prórroga en la final). Usar JSONB evita crear múltiples tablas polimórficas o múltiples columnas `answer_X` que queden nulas el 90% del tiempo. Para el Audit Log, permite guardar de forma nativa el payload completo modificado sin atarse a un esquema estricto.
- **Estrategia de bloqueo de predicciones (Locking)**: El requerimiento de no modificar predicciones después de iniciado el partido se manejará en la capa de aplicación (FastAPI). Las predicciones se rechazan si `NOW() > matches.match_date - polla_config.prediction_lock_minutes_before_match`. Delegar esto a la lógica de aplicación en lugar de hacerlo con un Trigger PL/pgSQL reduce la carga sobre la base de datos y mantiene la lógica de negocio centralizada.
- **Materialización del Leaderboard (`MATERIALIZED VIEW`)**: El cálculo de posiciones requiere leer tablas grandes y usar funciones de ventana (`RANK() OVER`), uniones múltiples y sumatorias condicionales (`COUNT FILTER`). Calcular esto en cada visita al "Dashboard" derrumbaría el rendimiento en un torneo de alta concurrencia. La vista materializada garantiza tiempos de respuesta O(1), y puede ser recargada (`REFRESH MATERIALIZED VIEW CONCURRENTLY`) únicamente de forma asíncrona mediante un evento disparado cuando un Admin actualiza el marcador de un partido.
- **Tipos Enumerados (`ENUM`) Nativos**: Usar ENUMs de PostgreSQL para estados (`scheduled`, `live`, `finished`), fases y posiciones, otorga validación a nivel de base de datos previniendo strings inválidos por errores en el backend, y ocupa menos espacio en disco que un VARCHAR estandarizado, mejorando la velocidad de los escaneos secuenciales y de indexación.
- **Foreign Keys (`RESTRICT` vs `CASCADE`)**: Las relaciones núcleo del torneo (ej. `matches` hacia `teams`) están en `ON DELETE RESTRICT` para asegurar que el borrado accidental de un equipo no desencadene el borrado en cascada de todo el historial de partidos. Sin embargo, los datos originados por usuarios (`match_predictions`) usan `ON DELETE CASCADE` hacia `users` para que la eliminación de una cuenta limpie instantáneamente su huella en la polla (cumplimiento GDPR y optimización de almacenamiento).
- **Escalabilidad a múltiples pollas**: Se usó el patrón Single-Row para `polla_config` (`CHECK id=1`). Si el proyecto evoluciona para soportar miles de pollas creadas por usuarios ("Private Leagues"), esta tabla simplemente quitará su `CHECK` para convertirse en `leagues`, y las tablas `users` y `match_predictions` referenciarán a dicha tabla con un `league_id` permitiendo tenant-isolation eficiente.
